# 基于生物启发优化的 SKILL.md 自进化方法

## 摘要

`SKILL.md` 是一种面向智能体系统的轻量能力描述文件。它通常包含能力名称、触发描述、适用边界、执行步骤、输出格式、外部引用说明等文本片段。随着智能体应用从单一提示词扩展到可组合能力体系，`SKILL.md` 的质量不再只是写作问题，而是一个可评测、可搜索、可演进的文本优化问题。本文提出一种基于生物启发优化的 `SKILL.md` 自进化框架：将 skill 文档表示为可变异、可组合、可评测的文本个体；将静态质量检查、触发评测和执行效果评测转化为结构化证据；再由进化策略依据证据生成候选文档，并对每个候选自动评测。数值分数只用于人类比较，不直接作为优化器的唯一目标。本文重点保留两个算法实例：基因算法（遗传算法）和粒子群优化。基因算法（遗传算法）适合把 `description`、`workflow`、`output`、`guardrails` 等 chunk 作为基因片段进行选择、交叉与变异；粒子群优化则适合把每个完整 `SKILL.md` 候选视为粒子，将不同 chunk 视作文本空间的坐标维度，使候选在自身最佳经验和全局最佳方向之间渐进收敛。该框架的价值在于把 skill 优化从“凭经验改提示词”转化为“基于评测证据的可追踪文本搜索过程”。

**关键词：** SKILL.md；自进化；生物启发优化；基因算法（遗传算法）；粒子群优化；智能体评测；提示工程

## 1. 引言

智能体系统中的 skill 不是普通文档。它既是用户意图和智能体能力之间的路由信号，也是能力执行过程中的行为约束。一个质量较差的 `SKILL.md` 可能导致两类问题：其一，该 skill 在应该触发时没有触发，或在不应该触发时误触发；其二，该 skill 触发后，输出缺少结构、边界或可核查性。传统提示词优化往往依赖人工经验：观察失败案例，手动修改描述，再凭感觉判断是否变好。这个过程难以复现，也难以解释每次修改带来的收益。

本文关注的问题是：如果把 `SKILL.md` 看作一种可演化文本工件，是否可以借鉴生物启发优化算法，将它放入一个“生成候选、评测候选、选择方向、继续演化”的闭环中？这里的“生物启发”并不意味着机械套用自然界隐喻，而是提取其中有工程价值的优化思想：种群、多候选、局部变异、片段重组、个体记忆、群体经验和逐步收敛。

在本文框架中，评测器不直接优化分数。分数用于人类阅读和候选比较，而算法消费的是结构化证据：静态检查发现了哪些缺陷，触发评测中有哪些 false positive 或 false negative，执行评测中哪些字段缺失或哪些行为不满足预期。这一点非常关键，因为单一总分会掩盖失败原因，也会诱导优化器追逐表面指标。一个好的 skill 自进化系统应当优化“失败证据背后的文本原因”，而不是优化一个抽象数字。

## 2. 问题定义

给定一个当前 skill 文档 \(S_0\)、一组评测器 \(E\)、一组评测集 \(D\)、一个进化方向 \(G\) 以及可选的进化环境文件 \(R\)，目标是在有限轮次内生成候选集合：

\[
C = \{S_{1,1}, S_{1,2}, ..., S_{t,k}\}
\]

其中每个候选 \(S_{t,k}\) 都是一个可下载、可独立运行的 `SKILL.md`。每个候选生成后都必须经过同一套评测器评测，得到结构化结果：

\[
E(S_{t,k}, D) \rightarrow \{F_{static}, F_{trigger}, F_{execution}, U\}
\]

其中：

- \(F_{static}\)：静态质量发现，例如 frontmatter 缺失、description 不清晰、缺少不适用场景、执行步骤不足、外部引用未说明。
- \(F_{trigger}\)：触发评测证据，例如应触发未触发、误触发、与其他 skill 描述重叠。
- \(F_{execution}\)：执行效果证据，例如缺少预期字段、输出不可核查、编造缺失信息。
- \(U\)：模型调用、token 用量、运行元信息等辅助记录。

优化器的目标不是最大化一个不可解释的总分，而是基于这些证据生成更好的文本候选。分数可以从证据汇总得到，但它只服务于人类比较、排序和筛选。

## 3. SKILL.md 的可进化表示

`SKILL.md` 可以被表示为一个由多个语义 chunk 组成的文本个体：

```text
S = {
  name,
  description,
  trigger_boundary,
  when_not_to_use,
  workflow,
  output_format,
  failure_handling,
  references
}
```

不同 chunk 在自进化中的作用不同。

`name` 和 `description` 主要影响路由和触发。它们决定智能体或路由器能否在用户输入、记忆、对话上下文以及其他 skill 描述同时存在时正确选择该 skill。

`trigger_boundary` 和 `when_not_to_use` 负责减少误触发。对 skill 而言，“什么时候不该用”与“什么时候该用”同样重要。缺少禁止场景会让 skill 在背景式提及、相邻任务或其他能力更适合的情况下被错误触发。

`workflow` 和 `output_format` 主要影响执行效果。它们让智能体知道应该如何处理输入、如何拆解任务、如何输出可核查结果。

`failure_handling` 处理缺失信息和不确定性。对于会议纪要、需求整理、数据分析等任务，skill 不应在信息缺失时编造负责人、截止时间或事实依据。

`references` 表示外部文件引用。若 `SKILL.md` 引用了 `references/`、`scripts/`、`assets/`、`templates/` 等文件，评测系统应将“引用是否存在、是否被说明、是否与正文一致”作为静态质量项之一。即便第一阶段只优化 `SKILL.md` 本体，外部文件是否被 skill 文本正确引用仍然影响可用性。

这种 chunk 化表示使生物启发算法可以在文本层面拥有“基因”“坐标”“局部位置”等可操作对象。

## 4. 评测证据驱动的自进化闭环

本文提出的自进化闭环包含六个阶段。

第一，输入或编辑当前 `SKILL.md`。平台不承担 skill 生命周期管理，也不是 skill 广场或下载站。它只负责轻量存储、评测、生成候选和下载满意版本。

第二，选择评测器和评测集。评测器至少包括静态质量检查、可触发性评测、单 skill 预期执行效果评测。agent 实际执行效果评测可以作为后续扩展，在没有真实 agent runner 的阶段保持独立但不强制接入。

第三，运行基线评测。系统对原始 skill 形成结构化证据，用作后续候选生成的初始反馈。

第四，进化策略生成候选。候选可以是链式生成，也可以是树形生成。链式生成每轮只产生一个候选，适合直接改进；树形生成每轮生成多个候选，适合探索不同方向。

第五，每个候选自动评测。候选生成后不能只展示文本 diff，而必须使用同一套评测器重新评测，形成可比较证据。

第六，用户比较候选并下载。平台不强制接受某个候选，也不管理版本生命周期。用户可以查看树形关系、每个候选的评测结果、相对原始 skill 的 diff，并下载自己满意的 `SKILL.md`。

## 5. 基因算法（遗传算法）实例：把 skill chunk 视作基因

基因算法（遗传算法）最早由 Holland 系统化提出，其核心思想是通过选择、交叉和变异，在候选种群中保留并重组高质量 building blocks。映射到 `SKILL.md` 优化时，完整 skill 文档是个体，语义 chunk 是基因。

### 5.1 表示

假设原始 skill 为：

```markdown
---
name: meeting-summary
description: Use this skill when the user wants help with meeting notes.
---

Summarize the meeting and list action items.
```

它至少包含两个基因：

- `descriptionOriginal`
- `instructionOriginal`

第一轮可以派生出多个候选：

- 候选 A：只变异 `description`，增强触发描述。
- 候选 B：保留原始 description，新增 `workflow`。
- 候选 C：保留原始 description，新增 `output_format`。

每个候选都作为完整 `SKILL.md` 评测，而不是只评测单个 chunk。

### 5.2 选择

选择不是简单选择最高总分，而是选择最有解释力的优势片段。例如候选 A 在触发评测中明显减少 false negative，则 A 的 `description` 可进入下一代；候选 B 在执行评测中改善步骤完整性，则 B 的 `workflow` 可进入下一代；候选 C 让输出字段更可核查，则 C 的 `output_format` 可进入下一代。

### 5.3 交叉

第二轮可以将 A 的 `description`、B 的 `workflow`、C 的 `output_format` 交叉为新候选：

```text
S2 = A.description + B.workflow + C.output_format
```

这类交叉的价值在于避免单一路径优化的局限：触发质量、执行步骤和输出结构通常由不同文本片段决定。遗传交叉允许系统把不同候选的局部优点拼成一个结构更完整的文档。

### 5.4 变异

变异用于针对失败证据进行小范围修复。例如：

- 如果评测发现 action item 缺少负责人字段，则变异 `output_format`，加入 `owner`。
- 如果评测发现缺少截止时间，则加入 `deadline`。
- 如果评测发现模型会编造信息，则加入 `failure_handling`，要求缺失信息标记为 `unspecified`。

变异应当是证据驱动的，而不是随机改写全文。对于 `SKILL.md` 这种短文本配置文件，过大幅度变异会破坏已有有效结构。

### 5.5 适用场景

基因算法（遗传算法）适用于以下情况：

- skill 有多个质量维度同时需要优化。
- 不同候选在不同 chunk 上各有优势。
- 用户希望看到清晰的派生、交叉、变异关系。
- 系统需要展示“哪些文本片段被保留，哪些片段被替换”。

它的主要风险是候选数量增长较快，且交叉后的文本可能出现风格不一致。因此系统需要在交叉后进行一致性整理，并限制每轮候选数。

## 6. 粒子群优化实例：把完整 skill 候选视为粒子

粒子群优化由 Kennedy 和 Eberhart 提出，原本用于连续函数优化，其核心思想是每个粒子在搜索空间中移动，同时参考自身历史最佳位置和群体全局最佳位置。映射到 `SKILL.md` 优化时，每个粒子不是单个 chunk，而是一个完整候选文件；chunk 是该粒子在文本空间中的坐标维度。

### 6.1 表示

一个粒子可以表示为：

```text
P_i = {
  description_position,
  workflow_position,
  output_position,
  guardrails_position,
  failure_handling_position
}
```

这些 position 不是数值坐标，而是文本状态。例如 `description_position` 可以是“粗略描述”“明确会议纪要触发”“加入 not-to-use 边界后的描述”等离散文本状态。

### 6.2 初始化

Round 0 只有原始 skill。Round 1 从原始 skill 派生出多个完整文件粒子：

- 粒子 A：关注 `description`，主要移动触发描述维度。
- 粒子 B：关注 `workflow/output`，主要移动执行路径和输出结构维度。
- 粒子 C：关注 `guardrails`，主要移动边界和禁止场景维度。

每个粒子都被独立评测，并形成自己的 pBest。若粒子 C 在综合证据上最稳定，它成为当前 gBest。

### 6.3 移动

Round 2 中，每个粒子不应简单向右线性演进，而应根据自身 pBest 和群体 gBest 调整位置：

```text
P_i(t+1) = move(P_i(t), pBest_i, gBest, evidence)
```

在文本空间中，移动可以解释为：

- 粒子 A 保留自身较好的 `description`，吸收 gBest 的 `guardrails`。
- 粒子 B 保留自身较好的 `workflow`，吸收 gBest 的 `description`。
- 粒子 C 作为 gBest，继续补充 `workflow` 和 `output`，扩大稳定优势。

这意味着粒子群优化不是把单个 chunk 当作粒子，而是让多个完整 `SKILL.md` 文件在不同 chunk 维度上逐步靠近更优区域。

### 6.4 收敛

随着轮次增加，粒子之间的差异应逐步减少。最终候选可能包含：

- 更准确的 description。
- 更明确的 when-not-to-use。
- 更完整的 workflow。
- 更可核查的 output format。
- 更稳健的 failure handling。

与基因算法（遗传算法）相比，粒子群优化通常产生更平滑的 diff。它不强调把不同父代片段显式交叉，而强调完整文件候选在文本空间中的渐进调参。

### 6.5 适用场景

粒子群优化适用于以下情况：

- 原始 skill 已有基本质量，不希望大幅重写。
- 用户希望看到多个候选文件如何逐步靠近共同方向。
- 优化目标更偏稳定性和渐进收敛，而非大规模结构重组。
- 评测证据能够区分个体最佳经验和全局最佳方向。

它的主要风险是早期探索不足。如果初始粒子太相似，群体会过早收敛到局部最优。因此初始化阶段应保证不同粒子关注不同 chunk。

## 7. 评测设计

一个有效的 skill 自进化平台需要把评测拆成独立问题。

### 7.1 静态质量检查

静态质量检查只依赖 `SKILL.md` 本身，适合发现结构问题：

- frontmatter 是否能解析。
- 是否包含 `name` 和 `description`。
- `description` 是否足够描述触发场景。
- 是否写明不适用或禁止使用场景。
- 是否包含清晰 workflow。
- 是否引用外部文件，且引用是否合理。

### 7.2 可触发性评测

可触发性评测回答：

> 给定用户输入、记忆和对话上下文，以及其他 skill 的名称和描述，当前 skill 是否应该被触发？

它应独立于执行效果评测。一个 skill 可能非常会执行，但不该被触发；也可能触发正确，但执行质量差。将二者混在一起会让优化器无法定位问题。

### 7.3 单 skill 预期执行效果评测

该评测默认 skill 已经触发，只判断按 `SKILL.md` 执行后是否满足预期结果。它关注：

- 是否覆盖用户任务。
- 是否输出必要字段。
- 是否保持事实依据。
- 是否处理缺失信息。
- 是否符合预期格式。

### 7.4 agent 实际执行效果评测

该评测需要真实 agent runner。早期系统可以保留数据结构和 UI，但不强行接入真实执行。它与单 skill 预期执行评测应保持分离，因为真实 agent 的表现还受工具调用、上下文管理、其他 skill 竞争等因素影响。

## 8. 系统实现建议

从工程角度，系统应包含以下模块：

- `Skill Store`：存储当前 `SKILL.md` 和实验快照。
- `Eval Set Store`：存储不同类型评测集。
- `Evaluator`：输出结构化证据和人类可读分数。
- `Evolution Strategy`：根据算法思想生成候选。
- `Candidate Graph`：记录候选之间的父子、交叉、移动、变异关系。
- `Diff Viewer`：显示每个候选相对原始 skill 或其他候选的差异。
- `Download`：允许用户拿走满意的 `SKILL.md`。

候选图是产品体验中的关键。基因算法（遗传算法）需要展示父代、子代、交叉线和变异线；粒子群优化需要展示每轮粒子位置、移动方向、pBest、gBest 和上轮 ghost 状态。只有把优化过程可视化，用户才能理解“算法思想如何落到文本变化上”。

## 9. 讨论

将生物启发优化用于 `SKILL.md` 的关键收益不在于算法名称本身，而在于它提供了一套组织搜索过程的语言。

基因算法（遗传算法）让用户理解“好文本片段可以被保留、交叉、变异”。这适合解释复杂文本结构如何逐步组合。

粒子群优化让用户理解“多个完整文件候选可以各自探索，然后参考自身经验和群体经验收敛”。这适合解释为什么某些优化不是一次性重写，而是逐步靠近稳定区域。

这两种算法都要求评测器输出足够细的证据。如果评测器只返回一个总分，算法只能盲目搜索；如果评测器返回失败类型、失败 case、具体字段缺陷和边界冲突，算法就能生成有针对性的文本变化。

## 10. 局限与未来工作

本文框架仍有局限。

第一，LLM 评测存在不确定性。需要通过固定 rubric、多次采样、人工抽检或回归评测集降低波动。

第二，文本搜索空间不是连续空间。粒子群优化中的“位置”和“速度”必须被重新解释为 chunk 状态和编辑方向，不能简单照搬数值公式。

第三，候选数量与成本需要控制。树形候选会迅速增加 LLM 调用成本，因此应限制轮次、候选数和评测集规模。

第四，真实 agent 执行评测尚需接入。只有接入真实工具调用和真实运行环境后，才能评估 skill 在完整 agent 系统中的效果。

第五，外部引用文件的语义一致性仍需加强。当前阶段可以先检查是否引用和是否说明，后续可以评估引用文件内容是否真的被 skill 正确使用。

## 11. 结论

本文提出了一种基于生物启发优化的 `SKILL.md` 自进化方法。该方法把 skill 文档看作可进化文本个体，把评测结果转化为结构化证据，并通过进化策略生成、评测和比较候选版本。基因算法（遗传算法）展示了如何把 skill chunk 作为基因进行选择、交叉和变异；粒子群优化展示了如何把完整候选文件作为粒子，在 pBest 与 gBest 的共同影响下逐步移动。该框架的核心不是让算法替代人类判断，而是让人类能够看到每一轮文本变化、评测证据和候选关系，从而更有把握地选择并下载更好的 `SKILL.md`。
